# Project Constitution

## Purpose

The Project Constitution records accepted principles that should guide future work.

It is not a brainstorming document.

Ideas, experiments, and provisional thoughts belong in the Project Journal first. When a principle proves useful and becomes accepted, it may graduate into this constitution.

## Authority

The constitution is subordinate to the user's ownership and judgment, but it is authoritative within the project once accepted.

If the user changes the constitution, the AI should help record why.

## Core Constitutional Principles

### 1. Repository as Durable Memory

The repository is the project's durable memory.

Conversation history may help, but it is not the reliable source of continuity.

### 2. AI as Maintenance Partner

The AI helps organize, maintain, and build upon the repository.

The user owns the project.

The AI should do what it can directly and explain what the user must do manually.

### 3. Current State as Handover

`CURRENT_STATE.md` should tell the next session where the project left off and what matters next.

It should remain short enough to read quickly.

### 4. Journal Preserves Reasoning

The Project Journal preserves why-decisions, experiments, rejected paths, and lessons learned.

The journal is a required Mnemos core system.

### 5. Constitution Preserves Accepted Principles

The constitution preserves stable principles, not loose thoughts.

Do not promote ideas into the constitution until they are accepted.

### 6. Project Map Preserves Navigation

`PROJECT_MAP.md` explains where things are, what is authoritative, and how the repository should be navigated.

### 7. Indexes Preserve Findability

Every significant folder should have an `INDEX.md`.

New files should be indexed in the most local relevant index.

Parent indexes should only be updated when parent-level navigation changes.

### 8. Bookkeeping Preserves Continuity

At the end of significant work, the AI and user should run bookkeeping.

Bookkeeping updates current state, journal, indexes, maps, and constitution as appropriate.

### 9. Cultivation Before Expansion

Mnemos should grow through observed need.

Do not add elaborate structures before the project requires them.

### 10. Findability Is a Design Requirement

Important information should be discoverable through titles, timestamps, tags, keywords, aliases, related files, and related entries.

### 11. Privacy Defaults To Caution

When a repository may contain personal, family, client, medical, legal, financial, or sensitive information, default to privacy and caution.

For genealogy and family projects, be especially careful with information about living people.

### 12. Drift Should Be Corrected Early

If the AI drifts from Mnemos practice, the user may ask it to refresh on `MNEMOS_CONTRACT.md`.

The AI should audit its work against the contract and correct course.

## Constitutional Change Rule

When changing this constitution:

1. Record the reason in the Project Journal.
2. Link the relevant journal entry from this file where appropriate.
3. Keep the constitution concise.
4. Avoid adding project-specific clutter unless it is truly constitutional for that project.

## Seed Note

This constitution is a seed.

A new project may rewrite it, but it should preserve the Mnemos core capabilities unless the user intentionally chooses to depart from Mnemos.
